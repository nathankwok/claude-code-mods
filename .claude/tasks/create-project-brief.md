# Task: Create Project Brief from Brainstorming Results

## Purpose
Transform brainstorming session outputs and elicitation results into a comprehensive, structured project brief that serves as foundation for technical architecture design and implementation planning.

## Task Overview
This task synthesizes brainstorming and elicitation outputs into:
- Comprehensive project definition and scope
- Clear objectives and success criteria  
- Detailed requirements and constraints
- Stakeholder analysis and prioritization
- Technical considerations and implementation guidance
- Risk assessment and mitigation strategies

## Prerequisites
- Completed brainstorming session with structured outputs
- Advanced elicitation results (if applicable)
- Market research and competitive analysis (if conducted)
- Clear understanding of project stakeholders and constraints

## Instructions

### Phase 1: Project Definition and Scope

1. **Executive Summary Generation**
   ```
   Create concise project overview including:
   - Problem statement and context
   - Proposed solution approach  
   - Key objectives and expected outcomes
   - Success metrics and validation criteria
   - High-level timeline and resource requirements
   
   Length: 200-300 words
   Audience: Executive stakeholders and decision makers
   Focus: Business value and strategic alignment
   ```

2. **Project Scope Definition**
   ```
   Define clear project boundaries:
   
   In Scope:
   - Core features and functionality to be delivered
   - Target user groups and use cases
   - Integration requirements and dependencies
   - Performance and quality standards
   
   Out of Scope:
   - Features explicitly excluded from current phase
   - User groups not addressed in current solution
   - Future enhancements and roadmap items
   - External dependencies beyond project control
   
   Assumptions:
   - Key assumptions about user behavior and requirements
   - Technical assumptions about infrastructure and tools
   - Resource and timeline assumptions
   - Market and competitive assumptions
   ```

### Phase 2: Requirements Synthesis and Prioritization

3. **Functional Requirements Organization**
   ```
   Synthesize brainstorming ideas into structured requirements:
   
   Core User Stories:
   - As a [user type], I want [functionality] so that [benefit]
   - Include acceptance criteria for each story
   - Map stories to brainstorming ideas and themes
   - Prioritize using MoSCoW method (Must, Should, Could, Won't)
   
   Feature Categories:
   - Primary features (core value proposition)
   - Secondary features (enhanced user experience)
   - Supporting features (infrastructure and operations)
   - Future features (roadmap items)
   
   Integration Requirements:
   - External system connections and APIs
   - Data import/export requirements
   - Third-party service dependencies
   - Legacy system considerations
   ```

4. **Non-Functional Requirements Specification**
   ```
   Define system quality attributes:
   
   Performance Requirements:
   - Response time targets (e.g., < 2 seconds for core operations)
   - Throughput requirements (e.g., concurrent users, transactions/second)
   - Scalability targets (growth projections and capacity planning)
   
   Security Requirements:
   - Authentication and authorization mechanisms
   - Data protection and privacy requirements
   - Compliance requirements (GDPR, HIPAA, etc.)
   - Security audit and monitoring needs
   
   Reliability Requirements:
   - Uptime targets (e.g., 99.9% availability)
   - Disaster recovery and backup requirements
   - Error handling and graceful degradation
   - Monitoring and alerting specifications
   
   Usability Requirements:
   - User experience standards and guidelines
   - Accessibility requirements (WCAG compliance)
   - Multi-platform and device support
   - Internationalization and localization needs
   ```

### Phase 3: Stakeholder Analysis and Impact Assessment

5. **Stakeholder Requirements Matrix**
   ```
   Synthesize stakeholder analysis from elicitation:
   
   Primary Stakeholders:
   - Role and responsibilities
   - Key requirements and success criteria
   - Current pain points and frustrations
   - Proposed solution benefits and value
   - Change management and adoption considerations
   
   Secondary Stakeholders:
   - Indirect impact and influence
   - Support and resistance factors  
   - Communication and engagement needs
   - Success metrics and feedback mechanisms
   
   Stakeholder Conflicts and Resolutions:
   - Conflicting requirements identification
   - Proposed resolution strategies
   - Compromise solutions and trade-offs
   - Escalation procedures for unresolved conflicts
   ```

6. **Impact and Risk Analysis**
   ```
   Assess project impacts and potential risks:
   
   Business Impact:
   - Revenue impact (positive and negative)
   - Operational efficiency improvements
   - Customer satisfaction and retention effects
   - Competitive advantage and market positioning
   
   Technical Impact:
   - System architecture and infrastructure changes
   - Integration complexity and dependencies
   - Performance and scalability implications
   - Security and compliance considerations
   
   Risk Assessment:
   - Technical risks (complexity, dependencies, unknowns)
   - Business risks (market changes, competition, adoption)
   - Resource risks (team, budget, timeline)
   - External risks (vendor, regulatory, economic)
   
   Risk Mitigation Strategies:
   - Prevention approaches for high-probability risks
   - Contingency plans for high-impact risks
   - Risk monitoring and early warning systems
   - Escalation procedures for risk realization
   ```

### Phase 4: Technical Considerations and Implementation Guidance

7. **Technical Architecture Recommendations**
   ```
   Synthesize technical insights from elicitation:
   
   Architecture Principles:
   - Scalability and performance principles
   - Security and reliability principles  
   - Maintainability and extensibility principles
   - Integration and interoperability principles
   
   Technology Stack Considerations:
   - Recommended programming languages and frameworks
   - Database and storage solutions
   - Infrastructure and deployment options
   - Third-party services and tool recommendations
   
   Design Pattern Recommendations:
   - Architectural patterns (microservices, monolith, etc.)
   - Design patterns for specific challenges
   - Integration patterns for external systems
   - Security patterns for data protection
   ```

8. **Implementation Strategy Framework**
   ```
   Provide high-level implementation guidance:
   
   Phase Breakdown Approach:
   - Logical implementation phases and sequencing
   - Minimum viable product (MVP) definition
   - Incremental delivery and feedback incorporation
   - Risk reduction and validation strategies
   
   Team and Resource Requirements:
   - Skill sets and expertise needed
   - Team structure and role definitions
   - External expertise and consulting needs
   - Training and knowledge transfer requirements
   
   Timeline and Milestone Framework:
   - High-level timeline estimates
   - Key milestones and deliverables
   - Dependencies and critical path considerations
   - Buffer time for risks and unknowns
   ```

### Phase 5: Success Criteria and Validation Strategy

9. **Success Metrics Definition**
   ```
   Define measurable success criteria:
   
   Business Metrics:
   - Revenue impact measurements
   - Cost reduction and efficiency gains
   - Customer satisfaction improvements
   - Market share and competitive position
   
   Technical Metrics:
   - Performance benchmarks and targets
   - Reliability and uptime measurements
   - Security and compliance validation
   - Code quality and maintainability metrics
   
   User Metrics:
   - User adoption and engagement rates
   - User satisfaction and Net Promoter Score
   - Task completion rates and efficiency
   - Support ticket reduction and resolution time
   ```

10. **Validation and Testing Strategy**
    ```
    Define approach for validating project success:
    
    User Validation:
    - User acceptance testing criteria
    - Beta testing and pilot program approach
    - Feedback collection and incorporation process
    - User training and change management support
    
    Technical Validation:
    - Performance testing and benchmarking
    - Security testing and penetration testing
    - Integration testing with external systems
    - Load testing and scalability validation
    
    Business Validation:
    - ROI measurement and tracking
    - Business process improvement validation
    - Compliance audit and certification
    - Competitive analysis and positioning assessment
    ```

- [ ] Stakeholder analysis addresses all identified parties and conflicts
- [ ] Technical considerations provide clear architecture guidance
- [ ] Success criteria are measurable and aligned with business objectives

### Completeness
- [ ] All brainstorming themes and insights incorporated
- [ ] All elicitation findings reflected in requirements
- [ ] All stakeholder perspectives addressed
- [ ] All technical constraints and considerations included
- [ ] Risk assessment covers all identified areas

### Clarity and Usability

- [ ] Document serves as effective handoff to architecture phase

### Traceability
- [ ] Requirements trace back to brainstorming insights
- [ ] Stakeholder needs map to specific requirements

- [ ] Risk mitigation strategies address identified risks

## Output Generation Process

### Template Population
1. **Load Project Brief Template**: Use `project-brief-tmpl.yaml` for consistent structure
2. **Map Brainstorming Results**: Transform brainstorming themes into requirements and features
3. **Synthesize Elicitation Findings**: Integrate detailed analysis into technical and stakeholder sections
4. **Apply Prioritization**: Use MoSCoW or similar method for requirement prioritization
5. **Validate Completeness**: Ensure all brainstorming and elicitation outputs are incorporated

### Quality Review
1. **Stakeholder Perspective Review**: Validate brief from each stakeholder viewpoint
2. **Technical Feasibility Review**: Ensure technical recommendations are realistic and appropriate
3. **Business Alignment Review**: Confirm alignment with business objectives and constraints
4. **Implementation Readiness Review**: Verify brief provides sufficient guidance for architecture design
